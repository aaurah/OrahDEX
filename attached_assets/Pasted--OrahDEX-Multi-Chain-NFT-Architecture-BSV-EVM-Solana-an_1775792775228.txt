🧩 OrahDEX Multi‑Chain NFT Architecture

(BSV + EVM + Solana + any future chain)

1. NFT Registry (Universal Keeper Index)

This is the heart.

A single registry that tracks NFTs across all chains, each with:

• chain (BSV, ETH, BNB, MATIC, ARB, BASE, SOL, etc.)
• contract_or_txid
• token_id_or_outpoint
• owner
• metadata_uri
• collection_id
• is_wrapped (true/false)
• native_chain


This registry becomes the canonical truth for the DEX.

How it updates:

• BSV Monitor → parses NFT transfers, mints, burns
• EVM Indexer → listens to ERC‑721/1155 events
• Solana Indexer → listens to Metaplex events
• Bridge Relayer → updates wrapped NFTs when moved cross‑chain


This gives you a single cross‑chain NFT universe.

---

2. NFT Listing Engine (DEX‑style orderbook for NFTs)

NFTs become first‑class markets, just like tokens.

Each NFT has:

A. Ask Listings

• Owner lists NFT for a fixed price
• Price can be in any supported token (BSV, USDT, ETH, etc.)
• Listing stored in your existing order system


B. Bids

• Anyone can place a bid
• Bid is locked funds (escrow or HTLC depending on chain)


C. Optional: Auctions

• Time‑boxed
• Keeper‑executed
• Settlement identical to fixed‑price but triggered at expiry


This uses your existing orderbook + portfolio + settlement logic.

---

3. Settlement Flows (Chain‑specific but unified)

A. BSV Native NFT Settlement

• Buyer signs payment
• Seller signs NFT transfer
• Keeper orchestrates atomicity
• BSV Monitor confirms
• Registry + portfolio update


B. EVM NFT Settlement

• Standard ERC‑721/1155 transfer
• Or wrapped NFT mint/burn if cross‑chain
• Settlement confirmed via EVM indexer


C. Solana NFT Settlement

• Metaplex transfer instruction
• Keeper verifies signature + block inclusion
• Registry updates


D. Cross‑Chain NFT Settlement

If buyer and seller are on different chains:

• NFT is locked on source chain
• Wrapped NFT minted on destination chain
• Or reverse (burn → unlock)


Your existing bridge relayer handles this.
