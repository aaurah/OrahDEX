🧠 What BSV contracts can enforce for cross‑chain trades

Each chain has a settlement contract.
On BSV, that contract can enforce:

1. Intent authenticity

• Verify the user’s signature (ECDSA over secp256k1)
• Verify the intent hash matches the signed payload


2. Trade correctness

• Check amount_in matches the UTXO value
• Check min_amount_out is respected
• Check deadline not expired
• Check nonce not reused


3. Double‑fill protection

• Track the intent hash inside the UTXO
• Mark it as spent when solver fills it


4. Solver settlement

• Solver must provide proof they already paid the user on the destination chain
• Or solver takes the risk (your preferred model)


This is exactly the same logic you apply on EVM, Solana, etc — just expressed in sCrypt.

🧩 Minimal BSV settlement contract for cross‑chain intents

This is a simplified version of what your per‑chain settlement contract looks like:

import {
  assert,
  ByteString,
  method,
  prop,
  SmartContract,
  Sig,
  PubKey,
  hash256,
} from 'scrypt-ts'

export class IntentSettlement extends SmartContract {

  @prop() intentHash: ByteString
  @prop() userPubKey: PubKey
  @prop() deadline: bigint

  constructor(intentHash: ByteString, userPubKey: PubKey, deadline: bigint) {
    super(...arguments)
    this.intentHash = intentHash
    this.userPubKey = userPubKey
    this.deadline = deadline
  }

  @method()
  public fill(sig: Sig, providedIntent: ByteString) {

    // 1. Check intent hash matches
    assert(hash256(providedIntent) == this.intentHash, 'Intent mismatch')

    // 2. Check signature from user
    assert(this.checkSig(sig, this.userPubKey), 'Bad signature')

    // 3. Check deadline
    assert(this.timeLock < this.deadline, 'Expired intent')

    // 4. If all good, solver can claim the locked funds
  }
}
