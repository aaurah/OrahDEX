1. What’s actually broken (from your notes + screenshot)

• Fake solvency for real wallets:• ensureSeedForAsset is auto-crediting any wallet when balance is insufficient.
• seedInitialBalances is seeding real EVM wallets with demo balances on first order.

• Ledger integrity breach:• A real wallet with 0.0058 ETH can place a 525 ETH sell order.
• Portfolio still shows the tiny real balance, while the order book pretends the huge order is valid.

• Error handling is sabotaged:• lockForOrder throws INSUFFICIENT_FUNDS, but the catch block logs and continues, so the order is still created.

• Client is complicit:• No synchronous guard on the form.
• No clear surfacing of server-side rejection.

2. Server-side fixes (must-do, non-negotiable)

A. Kill auto-seeding for real wallets

In ensureSeedForAsset:

• Add wallet source awareness (e.g. walletSource: 'demo' | 'orahdex' | 'external').
• Only allow auto-seed when:• walletSource === 'demo' (or whatever you call your sandbox wallet).

• For external / real EVM wallets:• Do not create or top up balances.
• If balance < required lock → throw INSUFFICIENT_FUNDS and let it bubble.



Direct rule: Only demo wallets are ever seeded. Real wallets must reflect chain reality.

B. Stop seeding real wallets on first order

In seedInitialBalances:

• Add the same walletSource check.
• If walletSource !== 'demo', return immediately—no balances created, no credits.
• Keep this function strictly for demo/sandbox flows.


C. Fix `lockForOrder` and error propagation

In the order placement flow (e.g. placeOrder / lockForOrder):

• Remove any try/catch that swallows INSUFFICIENT_FUNDS.
• If you must catch, then:• Re-throw for INSUFFICIENT_FUNDS (or return a typed error up the stack).
• Only handle expected, non-fatal cases (logging, metrics) without continuing the order.

• Ensure the order is not inserted into the DB if the lock fails.


Contract: No lock, no order. Ever.
