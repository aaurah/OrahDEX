1. Per‑pair routing config table

Table: routing_profiles

column	type	notes	
id	uuid	pk	
base_symbol	text	e.g. BSV	
quote_symbol	text	e.g. USDT	
max_slippage_bps	int	e.g. 150 = 1.5%	
min_fill_fraction	numeric	e.g. 0.9	
max_internal_size	numeric	optional: cap internal notional	
oracle_required	boolean	if true, fallback to LE when oracle stale	
enabled	boolean	feature flag per pair	


Usage in router:

• On getHybridRoute(pair, amount):• Load routing_profiles row for that pair; if none, use global defaults.
• Use max_slippage_bps and min_fill_fraction instead of hard‑coded constants.
• If oracle_required = true and oracle is stale/missing → route external.
• If max_internal_size set and amount > max_internal_size → route external.



This gives you per‑market risk control without touching code.

2. Explicit partial‑fill semantics

Define this once and reflect it in code + API docs:

• Option A (simple): all‑or‑nothing for swaps• If internal fill fraction < min_fill_fraction → treat as “no internal route”, go full LE.
• No partial internal + LE split.

• Option B (advanced): split routing• Internal fills up to available depth.
• Remaining size routed to LE.
• User sees one “swap”, but backend executes two legs.



Given your current stage, I’d start with Option A (all‑or‑nothing) and design Option B now so you don’t paint yourself into a corner.

3. Split routing design (Option B)

Backend flow (conceptual):

1. Walk orderbook, compute:• internalFillAmount
• internalVWAP

2. If internalFillAmount > 0:• Execute internal leg for internalFillAmount.

3. For remaining = requestedAmount - internalFillAmount:• If remaining > 0:• Get LE quote for remaining.
• Execute LE leg.


4. Return a composite response:
