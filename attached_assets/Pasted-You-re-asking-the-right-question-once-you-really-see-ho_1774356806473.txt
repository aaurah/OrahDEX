You’re asking the right question—once you really see how “connect wallet → show balance → trade → confirm” works end‑to‑end, DEXs stop feeling magical and start feeling mechanical.

I’ll walk you through the full flow on an EVM chain (Ethereum, BSC, etc.), because that’s the clearest mental model. The same principles adapt to others.

---

1. What a wallet actually is

• Externally Owned Account (EOA):
A wallet is just:• Private key → kept secret
• Public address → derived from the key

• The wallet software (MetaMask, Phantom EVM, custom app) does three things:• Stores keys (or connects to a hardware wallet)
• Builds & signs transactions
• Talks to a node via JSON‑RPC (e.g., eth_getBalance, eth_sendRawTransaction)  web3.js



The DEX never sees your private key. It only talks to the wallet and to the chain.

---

2. “Connect wallet” – what actually happens

When you click Connect Wallet on a DEX:

1. Dapp loads a provider• Browser wallet injects window.ethereum (MetaMask style), or
• Mobile uses WalletConnect (QR / deep link to a wallet app)

2. Dapp requests accounts• It calls something like:• ethereum.request({ method: 'eth_requestAccounts' })

• Wallet pops a UI:• “Allow this site to view your address and suggest transactions?”

• User approves → wallet returns one or more addresses.

3. Dapp stores session state• Saves:• currentAddress
• currentChainId

• Subscribes to events:• accountsChanged
• chainChanged




From this moment, the DEX can read your balances and propose transactions, but only the wallet can sign them.

---

3. How the DEX shows your balances

Once connected, the DEX needs:

• Native balance (ETH, BNB, etc.)
• Token balances (ERC‑20s like USDC, WETH, etc.)


3.1 Native balance

The DEX (via the provider) calls:

• eth_getBalance(address, "latest")  Web3.py


This returns the balance in wei.
The frontend converts it to ETH and displays it.

3.2 ERC‑20 token balances

For each token:

1. The DEX knows the token contract address and ABI.
2. It calls the contract’s balanceOf(address) function via eth_call.  DEV Community
3. The contract returns the token balance (in smallest units).
4. Frontend formats it using the token’s decimals().


This is read‑only—no signature needed.

3.3 Periodic updates

To keep balances fresh:

• Polling:• Every X seconds, re‑call eth_getBalance and balanceOf.

• Event‑driven:• Subscribe to Transfer events for relevant tokens and update when your address appears.

• After actions:• After a swap/approval, force a refresh once the tx is confirmed.



---

4. How a trade works (DEX side)

Let’s use a Uniswap‑style AMM as the mental model.

4.1 Two key contracts

• Router contract• User interacts with this.
• Functions like swapExactTokensForTokens, swapExactETHForTokens, etc.

• Pair/Pool contract• Holds token reserves.
• Implements the AMM math (x * y = k).



The DEX frontend:

• Knows the router address + ABI.
• Knows the pool addresses or can ask the router for the best path.


---

5. The approval step (ERC‑20 logic)

Before the router can move your tokens, it needs allowance.

5.1 How allowance works

ERC‑20 has:

• approve(spender, amount)
• allowance(owner, spender)


Flow:

1. DEX checks allowance(user, routerAddress) via eth_call.
2. If allowance < needed amount:• It asks you to sign an approval transaction:• approve(routerAddress, amount)


3. Wallet:• Builds a transaction:• to = tokenContract
• data = approve(...)

• Shows gas estimate, etc.
• You confirm → wallet signs and broadcasts via eth_sendRawTransaction.



Once mined, the router can move up to that amount of your tokens.

---

6. Building and sending the swap transaction

When you click Swap:

1. DEX computes expected output• Reads pool reserves via getReserves().
• Applies AMM formula:\Delta y = \frac{y \cdot \Delta x \cdot (1 - fee)}{x + \Delta x \cdot (1 - fee)}

• Applies slippage tolerance to compute amountOutMin.

2. DEX prepares transaction data• Chooses the right router function, e.g.:• swapExactTokensForTokens(amountIn, amountOutMin, path, to, deadline)

• Encodes it into data.

3. DEX calls eth_sendTransaction via provider• to = routerAddress
• data = encodedFunctionCall
• value = 0 (unless swapping native token)
• Wallet intercepts this request.

4. Wallet shows confirmation• From: your address
• To: router contract
• Data: decoded as “Swap X for Y”
• Gas estimate, gas price/priority fee
• You approve or reject.

5. Wallet signs & broadcasts• Signs with your private key.
• Sends raw tx to node via eth_sendRawTransaction.



---

7. Pending → mined → confirmed

Once broadcast:

1. Pending state• DEX can:• Show tx hash
• Link to block explorer

• It polls eth_getTransactionReceipt(txHash) until non‑null.

2. Mined (1 confirmation)• Receipt appears with:• status (1 = success, 0 = revert)
• blockNumber
• logs (events emitted by router/pair contracts)


3. Confirmations• Confirmations = currentBlock - tx.blockNumber + 1
• DEX might:• Treat 1 confirmation as “done” for UX
• Show “1/12 confirmations” for extra safety


4. Reading swap result from logs• DEX parses:• Swap events from the pair contract
• Transfer events from token contracts

• Confirms:• How much token in/out
• That to address matches the user


5. Balance refresh• After success:• Re‑query eth_getBalance and balanceOf for the user.
• Update UI.




---

8. Time‑to‑time confirmations & state sync

To keep the UI honest over time:

• Block subscription or polling• On each new block:• Update block number
• Optionally re‑fetch balances if:• User has recent activity
• Or you detect relevant events



• Transaction watcher• Maintain a list of “in‑flight” txs per user.
• For each:• Poll getTransactionReceipt
• Once confirmed, mark as done and refresh balances.


• Reorg handling (advanced)• Very rare, but:• If a block is reorged out, a previously “confirmed” tx can disappear.
• For critical apps, wait N confirmations before treating as final.




---

9. Putting it all together – end‑to‑end flow

From wallet connect to confirmed trade:

1. User clicks Connect Wallet
2. DEX requests accounts → wallet returns address
3. DEX:• Reads native + token balances via RPC/contract calls

4. User sets trade (token A → token B, amount, slippage)
5. DEX checks allowance:• If needed, user signs approve tx

6. User clicks Swap:• DEX builds router call
• Wallet shows tx → user confirms

7. Tx is broadcast:• DEX shows pending state
• Polls for receipt

8. Once mined:• DEX parses logs
• Updates balances
• Shows “Swap successful” with final amounts



Everything is just:

• Wallet = keys + signing + RPC bridge
• DEX = UI + contract calls + math
• Chain = source of truth (balances, events, confirmations)
