🧩 What’s left to make this subsystem perfect

You’re 95% of the way to a fully sovereign, horizontally scalable, multi‑instance HTLC engine.
Here are the last 3 refinements that would make it bulletproof:

---

1. Add a “terminalAt” timestamp to htlc_registry

Right now, terminal states (CLAIMED, REFUNDED) are stored, but you don’t know when they became terminal unless you query events.

Add:

terminalAt TIMESTAMP NULL


Set it when:

• toStatus ∈ {CLAIMED, REFUNDED}


This gives you:

• Fast queries for “recently completed HTLCs”
• Ability to purge old rows
• Keeper scoring windows (“HTLCs resolved in last 30 days”)


---

2. Add a “nextCheckAt” field for scalable polling

Right now the watcher polls all active HTLCs every 90 seconds.

That’s fine for dozens.
But when you have thousands, you’ll want:

nextCheckAt TIMESTAMP NOT NULL DEFAULT NOW()


Then:

• When HTLC is far from locktime → check less often
• When HTLC is near locktime → check more often
• When HTLC is expired → check aggressively until resolved


This gives you:

• Adaptive load
• Zero wasted polling
• Natural sharding if you ever run multiple watchers


---

3. Add a “keeperActions” table for future scoring

You already have:

• HTLC registry
• HTLC events
• Relayer events endpoint
• Keeper UI panel


The missing piece is the Keeper action log:

keeper_actions (
  id SERIAL PRIMARY KEY,
  keeperAddress TEXT NOT NULL,
  htlcAddress TEXT NOT NULL,
  action TEXT NOT NULL, -- e.g. "CLAIMED", "REFUNDED", "OBSERVED"
  txid TEXT,
  createdAt TIMESTAMP DEFAULT NOW()
)


This unlocks:

• Keeper reputation
• Keeper rewards
• Keeper slashing
• Keeper leaderboards
• Keeper “lineage” (mythic narrative)


This is where the ceremonial layer meets the technical layer.
